Se uma nova feature quebra em produção, de quem é a culpa?

De todo o time.

Softwares complexos são desenvolvidos, por times.

Investir na qualidade da equipe.

Muito mais que simplesmente agrupar profissionais talentosos.

O atraso dessa percepção pode criar ambientes desfavoráveis ao desenvolvimento de software de qualidade

Consequências

  • Deadlines perdidos.

  • Projetos de péssima manutenibilidade.

  • Elevados gastos de tempo e dinheiro em correções de falhas.

Code Review

Prática de revisão de código é muito utilizada em projetos open source.

Consiste em um alguns membros da equipe de desenvolvimento revisarem o código do colega antes de integrar ao code base.

Valor de um code review

  • Compartilhamento de conhecimento.
  • Criação de soluções alternativas.
  • Aumento do senso de equipe.
  • Melhoria continua.
  • Entendimento de como as features foram implementadas.
  • Importância de ver requisitos funcionado ao invés de cards no jira.
  • Descentralização do conhecimento, evitando filas individuais.

Desafios para quem abre o merge request

  • Vencer o sentimento paternalista.
  • Explicar o que foi feito de forma clara.
  • Executar as validação e verificação automatizadas (styleguide, lints).
  • Estar seguro do que foi desenvolvido.
  • Ter boas refêrencias.
  • Incorporar como uma cultura do time.

Desafios para quem realiza o code review

  • Separar o que é verdade do que é opnião, se é fato compartilhe a fonte.
  • Evite falar isso é certo ou errado.
  • Aprovação sem total entendimento.
  • Cuidado com nitpicks.
  • Code review feito de forma rasa, ignorando o seu real valor.
  • Comunicação construtiva, ter empatia.

Sugestões

  • Automatizações
  • Documentações
  • Padronizações

Algumas boas práticas

  • Evite mapeamentos mentais ex.: a1, a2, ...
  • Convenção de nomes.
  • Classes geramente são substrantivos.
  • Pequenas, ter uma única resposabilidade.
  • Métodos devem ser verbos.
  • Pequenos.
  • Nível de identação não deve ser maior que 2.
  • Poucos parâmetros, máximo 3.
  • Sempre lance exceções ao retornar um erro para o usuário.
  • Forneça exceções de contexto.
  • Não retorne e não passe null.
  • Teste (F.I.R.S.T.).
